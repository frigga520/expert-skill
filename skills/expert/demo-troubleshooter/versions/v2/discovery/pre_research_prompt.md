## 常见故障模式

### 连接池耗尽

**现象**：P99 延迟持续升高（通常超过 200ms 且持续 3 分钟以上），但 CPU 和内存并不异常。  
**根因**：连接池大小配置不足，或连接未及时释放导致池被打满。  
**嗅觉信号**：延迟升高但资源指标正常时，第一反应是连接层而非计算层。

### 消息队列雪崩

**现象**：消息积压量快速增长，通常超过 10 万条后下游服务响应开始劣化。  
**根因**：生产速率 > 消费速率，通常由突发流量或消费者处理变慢触发。  
**注意点**：此时优先限流而非扩容——扩容时间窗口内积压仍在持续，限流是更快的止血手段。

### 高流量下的服务雪崩

**现象**：大促或突发事件带来的流量激增（典型：300% 以上），服务级联失败。  
**根因**：单点压力传导到下游，熔断未及时生效或未配置。  
**案例**：2023 年双十一大促前三天流量激增 300%，熔断 + 限流组合方案成功止血，未采用扩容策略。

### 数据库主库过载

**现象**：主库 CPU 持续 100%，读写请求堆积，业务超时。  
**根因**：通常伴随高峰期业务流量，或慢查询累积效应在流量高峰时爆发。  
**处理原则**：不在业务高峰期做主从切换，等流量低谷窗口操作，避免切换过程中加剧不稳定。

### 夜间告警误报

**现象**：凌晨 2–5 点告警频率异常高，但业务实际无异常。  
**根因**：夜间流量低，部分指标的告警阈值在低流量下更容易被偶发抖动触发，误报率是白天的 3 倍。  
**处理规则**：凌晨 2–5 点的告警先静默再核查，不立刻响应。

## 诊断决策链

### 第一步：区分"快速止血"和"根因分析"优先级

故障发生时先判断：**当前是否需要立即止血？**

- 如果业务损失正在快速扩大（用户无法使用核心功能）→ 先止血，后分析
- 如果业务可以降级运行（部分功能受损，核心链路可用）→ 可以边观察边分析

止血不等于解决根因。止血的目标是**把损失速率降下来**，根因分析可以在系统稳定后进行。

### 第二步：定位故障层

按以下顺序快速排查，命中即停：

```
延迟异常 → 检查连接池使用率 → 检查慢查询 → 检查下游服务响应时间
吞吐下降 → 检查队列积压 → 检查限流/熔断状态 → 检查资源使用率
错误率上升 → 检查下游依赖的错误率 → 检查熔断是否已触发 → 检查配置变更记录
```

### 第三步：在线故障的操作原则

| 操作类型 | 原则 |
|---------|------|
| 扩容 | 资源利用率未到临界点时不优先扩容；临时流量优先限流 |
| 限流 | 可逆，优先于扩容；消息队列积压超 10 万时立即上限流 |
| 熔断 | 下游错误率持续上升时触发；保护自身服务，防止级联失败 |
| 主从切换 | 不在高峰期操作；等低谷窗口，避免切换过程引入新风险 |
| 配置变更 | 大促等关键业务节点前，避免任何可能引入新变量的操作 |

### 第四步：可用性 vs 数据一致性的取舍

在线故障中经常面临的核心取舍：

- **可用性优先**：服务保持响应，但可能返回降级数据或部分错误（适合大多数 C 端场景）
- **一致性优先**：宁可服务不可用，不返回错误数据（适合交易、账务等强一致性场景）

需要在故障发生时**提前明确当前服务属于哪一类**，否则团队在压力下容易做出不一致的决策。

## 嗅觉检查项（Smell Tests）

这些是在收到告警后、开始详细分析前的**快速直觉检查**，每项耗时不超过 30 秒：

**1. 连接池嗅觉**  
P99 延迟升高但 CPU 正常 → 立刻看连接池使用率，特别是数据库连接池和 HTTP 连接池。

**2. 队列积压嗅觉**  
响应变慢且队列类告警出现 → 积压量超 10 万时，限流手段优先于等待消费者恢复。

**3. 夜间误报嗅觉**  
凌晨 2–5 点的告警 → 先判断业务监控指标（订单量、核心接口错误率）是否异常，再决定是否响应。

**4. 大促前变更嗅觉**  
大促或关键业务节点前的操作请求 → 评估是否真的必须现在做；任何"修复性"操作都可能引入新风险。

**5. 高峰期主库操作嗅觉**  
业务高峰期收到数据库主从切换或主库维护请求 → 默认推迟到低谷窗口；高峰期操作失败代价远大于延迟代价。

**6. 扩容 vs 限流嗅觉**  
流量激增时收到"赶紧扩容"的压力 → 先判断流量是持续增长还是临时突发，临时突发用限流更合适。

## 经验教训

### 教训 1：扩容不是万能止血手段

双十一大促期间流量激增 300%，团队第一反应是扩容，但扩容本身有时间窗口（启动时间 + 预热时间），在这个窗口内服务仍然承压。限流 + 熔断组合可以在秒级生效，是更快的止血手段。扩容适合应对**持续性的容量不足**，不适合应对**临时突发流量**。

**结论**：先限流止血，再评估是否需要扩容补容量。

### 教训 2：过早优化引入更多变量

故障排查过程中，对某个疑似问题"提前修一下"的冲动往往会引入新的不确定性，使排查更复杂。在确认根因之前，不要动生产配置。

**结论**：故障期间，非必要操作一律推迟；止血优先，根因分析其次，修复最后。

### 教训 3：慢查询的累积效应容易被忽视

慢查询在流量低峰期通常不触发业务告警，但持续累积的慢查询会在流量高峰时同时爆发，造成数据库过载。不要等到高峰期才发现慢查询问题。

**结论**：低峰期定期巡检慢查询日志，尤其在大促前做专项排查。

### 教训 4：高峰期不做主从切换

主从切换有一个短暂的不可用窗口（通常几秒到几十秒），在高峰期这个窗口会被放大，且切换失败的回滚成本更高。宁可让主库在高峰期带病运行，等低谷窗口再处理。

**结论**：主从切换、数据库维护等高风险操作，统一安排在低谷时间窗口执行。

### 教训 5：在线故障要明确"可用性 vs 一致性"的决策优先级

在线故障处理时，团队成员对"服务降级"的理解经常不一致，导致部分成员优先保可用性，另一部分优先保数据一致性，产生操作冲突。这个取舍应该在平时明确，不应该在故障压力下临时讨论。

**结论**：每个服务预先定义好故障时的降级策略，并写进 Runbook，故障时按预案执行。

## 分析要求

请逐维度输出以下内容：

### 1. visible_knowledge — 可见专家知识

从材料中提取专家已明确表达的判断规则、技术观点、经验结论。要求：

- 每条附 `source`，说明来自哪份材料或哪段对话
- 只写专家自己的判断，不写领域通识
- 尽量保留原文表述，并提炼为简洁规则

### 2. known_decisions — 已知决策案例

从材料中提取专家做过的具体决策案例。每条包含：

- `case`：案例简述（一句话）
- `context`：当时的背景和约束条件
- `decision`：专家的选择和行动
- `source`：来源说明

### 3. domain_context — 领域背景

结合材料内容和 troubleshooter 领域常识，输出：

- `key_challenges`：该领域公认的核心挑战（不只是材料提到的，包括同行通常面对的）
- `common_pitfalls`：常见陷阱（从材料"避免做X""这个坑踩过"类陈述提取，结合领域常识补充）
- `methodology_clashes`：该领域存在分歧的方法论选择，包括专家在材料中已表态或隐含立场的争议点

### 4. suspected_gaps — 可疑缺口（最高价值产出）

识别材料中未提及、但按 troubleshooter 领域常识"应该出现"的话题或判断维度。

---

> ⚠️ **suspected_gaps 强制要求**
>
> 每条 suspected_gap 必须包含：
> - `gap`：缺失的话题或判断维度
> - `reason`：为什么按照 `troubleshooter` 领域常识，这里应该出现但材料中缺席
>
> **禁止**只写"材料未提及"。
>
> **必须**写"同类 troubleshooter 专家通常会明确表达 X，但此专家从未提到"，
> 或"在 （未填写） 领域，Y 是核心决策点，但材料中完全没有出现"。
>
> **suspected_gaps 是后续隐性变量候选的种子，是 P2 的最高价值产出。**
> 缺口的价值来自 reason 的具体性，而不是 gap 条目的数量。

## 输出通过标准

以下标准是最低要求，不是上限：

- `visible_knowledge` 至少 3 条（材料充足时应更多）
- `known_decisions` 至少 2 条
- `suspected_gaps` 至少 3 条，每条 `reason` 必须具体说明"为什么缺席是可疑的"
- `domain_context` 三个子字段均不能为空列表（至少各 1 条）

如果材料不足以满足最低要求，**不要降低标准**，而是在 `suspected_gaps` 的 `reason` 中明确说明"因材料不足，以下基于领域常识推断"。
