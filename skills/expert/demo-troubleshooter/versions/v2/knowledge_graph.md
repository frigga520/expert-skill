# frigga — 知识图谱

## 专长类型: Troubleshooter

## 隐性变量节点

| 变量 | 置信度 | 来源三联体 |
|------|--------|-----------|
| 熔断触发三阶段时间窗口模型：3 分钟以内判断是否为瞬时抖动；5 分钟左右必须进入熔断评估；8 分钟无恢复迹象则必须触发。阈值来源：一次下游依赖抖动事故（线程池堆积 + 重试放大，等到 10 分钟才熔断，恢复时间被拉长） | high | tg_003 |
| 熔断触发时的二次确认信号：不依赖单一错误率数字，而是同时校验（1）错误率有无下降趋势；（2）自身服务是否被拖垮（线程池排队、P99 延迟上升）。两个信号决定是全量熔断、分层熔断还是继续带熔断开关待命 | high | tg_003 |
| CPU 资源水位两阶段阈值：80%（评估线，来自压测——P99 开始对流量波动变敏感，安全余量明显下降）和 85%（动作线，来自线上事故校准——超过该值叠加缓存 miss/GC/下游抖动时系统容易从'能扛'突然进入排队放大）。两个数字独立来源、含义不同 | high | tg_001 |
| 扩容决策还受 CPU 增长斜率影响：高斜率（每 5 分钟涨 3-5 个点）时不等待直接扩容；低斜率（每 10 分钟涨 1 个点）时可先扩一小批观察效果。斜率用于区分'容量窗口紧迫度' | medium | tg_001 |
| 可逆性偏好被效果差距打破的三段阈值：效果差 <10 个点时选可逆（引入不可逆风险不值得）；15-20 个点时重新评估（可逆方案可能只是延缓事故不是止血）；>20 个点且故障仍在扩大时切换到不可逆方案为主线。阈值来源：一次队列积压事故中坚持限流结果只是把'快速恶化'变成'慢速恶化' | high | tg_004 |
| 慢查询介入阈值双维度模型：P99 延迟升幅（用户感知线 2x，来自业务告警联动监测）和持续时间（毛刺过滤线 10 分钟，跨越两三个观察窗口才是真实问题）。两个维度均需满足才进入介入线，单一维度不触发 | high | tg_002 |

| 优先规则 | 从属规则 | 触发条件 |
|---------|---------|---------|
| rule_A |  | 当硬死线（如 10 分钟内恢复）物理上排除了效果更好的不可逆方案时 |
| rule_A（条件性胜出） |  | 当 P99 < 2x 时，大促临近优先；当 P99 > 2x 且有联动信号时，技术风险处理重新优先 |
| rule_A（条件性胜出） |  | 当降级响应已在 Runbook 中定义为安全时止血优先；当为写链路或降级可能被误判为成功时一致性优先 |
| 取决于流量持续性 |  | 监控明确显示持续增长且 CPU > 85% 时扩容主线；临时突发或 CPU < 85% 时限流主线 |

## 边界条件

| 边界内容 | 适用域 | 失效域 |
|---------|-------|-------|
| 熔断时间阶梯：3min 判断抖动，5min 进入评估，8min 无恢复必须触发 | 下游错误率持续上升场景，主要针对非核心链路或已定义降级语义的链路 | 强一致性链路（交易、支付、库存），降级安全前提不满足时该阈值不直接适用 |
| 降级响应安全判断：事前定义 70% 为主，现场校验 30% 为辅；无预案时默认明确失败 | 所有需要熔断触发降级的服务 | 无历史演练、无 Runbook 定义、降级与真实成功无区分标记的场景 |
| CPU 80% 为扩容评估线（压测来源），85% 为扩容动作线（事故来源） | 标准无状态服务，CPU 密集型处理节点 | I/O 密集型或内存密集型服务（CPU 不是性能瓶颈），以及有特定 GC 特征的 JVM 应用 |
| 流量持续性决定限流/扩容主从关系切换（持续增长→扩容主线；临时突发→限流主线） | 具备流量趋势监控的服务，CPU > 85% 场景 | 无法区分流量持续性的场景（无监控或流量模式不确定），此时默认限流保护 |
| 可逆性切换阈值：效果差 <10 点选可逆，15-20 点重新评估，>20 点切不可逆（前提：故障仍在扩大） | 两个方案止血效果有量化差距的故障响应决策 | 效果无法量化或两方案时间窗口差异极大时，该阈值不直接适用 |
| 慢查询介入阈值：P99 2x 为用户感知线，持续 10 分钟为毛刺过滤线，两者均满足才进入介入线 | 数据库慢查询监控，以 P99 尾延迟为主要观测指标的场景 | P99 绝对值本身很低（如平均 10ms，2x 仍为 20ms）时，倍率指标可能误导；批量报表类查询（慢但不影响在线 P99） |
| 大促前操作窗口三级：30min 只做止血开关，2h 低风险清理，6h+ 已验证修复（必须留 1-2h 观察窗口） | 有明确业务节点（大促、发布、高峰期）前的技术操作决策 | 无可预期业务节点的持续高流量场景，大促前三级框架不适用 |
