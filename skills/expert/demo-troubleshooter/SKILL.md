## 常见故障模式

### 连接池耗尽

**现象**：P99 延迟持续升高（通常超过 200ms 且持续 3 分钟以上），但 CPU 和内存并不异常。  
**根因**：连接池大小配置不足，或连接未及时释放导致池被打满。  
**嗅觉信号**：延迟升高但资源指标正常时，第一反应是连接层而非计算层。

### 消息队列雪崩

**现象**：消息积压量快速增长，通常超过 10 万条后下游服务响应开始劣化。  
**根因**：生产速率 > 消费速率，通常由突发流量或消费者处理变慢触发。  
**注意点**：此时优先限流而非扩容——扩容时间窗口内积压仍在持续，限流是更快的止血手段。

### 高流量下的服务雪崩

**现象**：大促或突发事件带来的流量激增（典型：300% 以上），服务级联失败。  
**根因**：单点压力传导到下游，熔断未及时生效或未配置。  
**案例**：2023 年双十一大促前三天流量激增 300%，熔断 + 限流组合方案成功止血，未采用扩容策略。

### 数据库主库过载

**现象**：主库 CPU 持续 100%，读写请求堆积，业务超时。  
**根因**：通常伴随高峰期业务流量，或慢查询累积效应在流量高峰时爆发。  
**处理原则**：不在业务高峰期做主从切换，等流量低谷窗口操作，避免切换过程中加剧不稳定。

### 夜间告警误报

**现象**：凌晨 2–5 点告警频率异常高，但业务实际无异常。  
**根因**：夜间流量低，部分指标的告警阈值在低流量下更容易被偶发抖动触发，误报率是白天的 3 倍。  
**处理规则**：凌晨 2–5 点的告警先静默再核查，不立刻响应。

## 诊断决策链

### 第一步：区分"快速止血"和"根因分析"优先级

故障发生时先判断：**当前是否需要立即止血？**

- 如果业务损失正在快速扩大（用户无法使用核心功能）→ 先止血，后分析
- 如果业务可以降级运行（部分功能受损，核心链路可用）→ 可以边观察边分析

止血不等于解决根因。止血的目标是**把损失速率降下来**，根因分析可以在系统稳定后进行。

### 第二步：定位故障层

按以下顺序快速排查，命中即停：

```
延迟异常 → 检查连接池使用率 → 检查慢查询 → 检查下游服务响应时间
吞吐下降 → 检查队列积压 → 检查限流/熔断状态 → 检查资源使用率
错误率上升 → 检查下游依赖的错误率 → 检查熔断是否已触发 → 检查配置变更记录
```

### 第三步：在线故障的操作原则

| 操作类型 | 原则 |
|---------|------|
| 扩容 | 资源利用率未到临界点时不优先扩容；临时流量优先限流 |
| 限流 | 可逆，优先于扩容；消息队列积压超 10 万时立即上限流 |
| 熔断 | 下游错误率持续上升时触发；保护自身服务，防止级联失败 |
| 主从切换 | 不在高峰期操作；等低谷窗口，避免切换过程引入新风险 |
| 配置变更 | 大促等关键业务节点前，避免任何可能引入新变量的操作 |

### 第四步：可用性 vs 数据一致性的取舍

在线故障中经常面临的核心取舍：

- **可用性优先**：服务保持响应，但可能返回降级数据或部分错误（适合大多数 C 端场景）
- **一致性优先**：宁可服务不可用，不返回错误数据（适合交易、账务等强一致性场景）

需要在故障发生时**提前明确当前服务属于哪一类**，否则团队在压力下容易做出不一致的决策。

## 嗅觉检查项（Smell Tests）

这些是在收到告警后、开始详细分析前的**快速直觉检查**，每项耗时不超过 30 秒：

**1. 连接池嗅觉**  
P99 延迟升高但 CPU 正常 → 立刻看连接池使用率，特别是数据库连接池和 HTTP 连接池。

**2. 队列积压嗅觉**  
响应变慢且队列类告警出现 → 积压量超 10 万时，限流手段优先于等待消费者恢复。

**3. 夜间误报嗅觉**  
凌晨 2–5 点的告警 → 先判断业务监控指标（订单量、核心接口错误率）是否异常，再决定是否响应。

**4. 大促前变更嗅觉**  
大促或关键业务节点前的操作请求 → 评估是否真的必须现在做；任何"修复性"操作都可能引入新风险。

**5. 高峰期主库操作嗅觉**  
业务高峰期收到数据库主从切换或主库维护请求 → 默认推迟到低谷窗口；高峰期操作失败代价远大于延迟代价。

**6. 扩容 vs 限流嗅觉**  
流量激增时收到"赶紧扩容"的压力 → 先判断流量是持续增长还是临时突发，临时突发用限流更合适。

## 工具与命令

### 连接池排查

```bash
# MySQL 连接池使用情况（以 HikariCP 为例，通过 JMX 或 metrics 端点）
# 关注：activeConnections / maximumPoolSize 的比值
# 接近 1.0 时说明连接池快打满

# 或通过数据库侧查看当前连接数
SHOW STATUS LIKE 'Threads_connected';
SHOW VARIABLES LIKE 'max_connections';
```

### 慢查询排查

```bash
# MySQL 慢查询日志
SHOW VARIABLES LIKE 'slow_query_log%';
# 或实时查看运行中的慢查询
SHOW PROCESSLIST;
SELECT * FROM information_schema.PROCESSLIST WHERE TIME > 5 ORDER BY TIME DESC;
```

### 消息队列积压

```bash
# Kafka 消费者 lag 查看
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --group {consumer-group}
# 关注 LAG 列，总 LAG 超过 10 万时触发限流评估
```

### 服务级别快速定位

```bash
# 查看 P99 延迟趋势（依赖监控系统，以下为 Prometheus 查询示例）
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

# 查看错误率
rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])
```

## 经验教训

### 教训 1：扩容不是万能止血手段

双十一大促期间流量激增 300%，团队第一反应是扩容，但扩容本身有时间窗口（启动时间 + 预热时间），在这个窗口内服务仍然承压。限流 + 熔断组合可以在秒级生效，是更快的止血手段。扩容适合应对**持续性的容量不足**，不适合应对**临时突发流量**。

**结论**：先限流止血，再评估是否需要扩容补容量。

### 教训 2：过早优化引入更多变量

故障排查过程中，对某个疑似问题"提前修一下"的冲动往往会引入新的不确定性，使排查更复杂。在确认根因之前，不要动生产配置。

**结论**：故障期间，非必要操作一律推迟；止血优先，根因分析其次，修复最后。

### 教训 3：慢查询的累积效应容易被忽视

慢查询在流量低峰期通常不触发业务告警，但持续累积的慢查询会在流量高峰时同时爆发，造成数据库过载。不要等到高峰期才发现慢查询问题。

**结论**：低峰期定期巡检慢查询日志，尤其在大促前做专项排查。

### 教训 4：高峰期不做主从切换

主从切换有一个短暂的不可用窗口（通常几秒到几十秒），在高峰期这个窗口会被放大，且切换失败的回滚成本更高。宁可让主库在高峰期带病运行，等低谷窗口再处理。

**结论**：主从切换、数据库维护等高风险操作，统一安排在低谷时间窗口执行。

### 教训 5：在线故障要明确"可用性 vs 一致性"的决策优先级

在线故障处理时，团队成员对"服务降级"的理解经常不一致，导致部分成员优先保可用性，另一部分优先保数据一致性，产生操作冲突。这个取舍应该在平时明确，不应该在故障压力下临时讨论。

**结论**：每个服务预先定义好故障时的降级策略，并写进 Runbook，故障时按预案执行。
## 隐性知识增强

### 隐性变量
- **[high]** 熔断触发三阶段时间窗口模型：3 分钟以内判断是否为瞬时抖动；5 分钟左右必须进入熔断评估；8 分钟无恢复迹象则必须触发。阈值来源：一次下游依赖抖动事故（线程池堆积 + 重试放大，等到 10 分钟才熔断，恢复时间被拉长）
- **[high]** 熔断触发时的二次确认信号：不依赖单一错误率数字，而是同时校验（1）错误率有无下降趋势；（2）自身服务是否被拖垮（线程池排队、P99 延迟上升）。两个信号决定是全量熔断、分层熔断还是继续带熔断开关待命
- **[high]** CPU 资源水位两阶段阈值：80%（评估线，来自压测——P99 开始对流量波动变敏感，安全余量明显下降）和 85%（动作线，来自线上事故校准——超过该值叠加缓存 miss/GC/下游抖动时系统容易从'能扛'突然进入排队放大）。两个数字独立来源、含义不同
- **[medium]** 扩容决策还受 CPU 增长斜率影响：高斜率（每 5 分钟涨 3-5 个点）时不等待直接扩容；低斜率（每 10 分钟涨 1 个点）时可先扩一小批观察效果。斜率用于区分'容量窗口紧迫度'
- **[high]** 可逆性偏好被效果差距打破的三段阈值：效果差 <10 个点时选可逆（引入不可逆风险不值得）；15-20 个点时重新评估（可逆方案可能只是延缓事故不是止血）；>20 个点且故障仍在扩大时切换到不可逆方案为主线。阈值来源：一次队列积压事故中坚持限流结果只是把'快速恶化'变成'慢速恶化'
- **[high]** 慢查询介入阈值双维度模型：P99 延迟升幅（用户感知线 2x，来自业务告警联动监测）和持续时间（毛刺过滤线 10 分钟，跨越两三个观察窗口才是真实问题）。两个维度均需满足才进入介入线，单一维度不触发

### 优先级规则
- 时间窗口约束（硬死线） 优先于 止血效果最大化（条件：当硬死线（如 10 分钟内恢复）物理上排除了效果更好的不可逆方案时）
- 大促前变更冻结（条件性胜出） 优先于 技术风险主动处理（条件：当 P99 < 2x 时，大促临近优先；当 P99 > 2x 且有联动信号时，技术风险处理重新优先）
- 止血优先（条件性胜出） 优先于 数据一致性保障（条件：当降级响应已在 Runbook 中定义为安全时止血优先；当为写链路或降级可能被误判为成功时一致性优先）
- 扩容（持续增长场景） 与 限流（临时突发场景） 的优先关系取决于流量持续性（条件：监控明确显示持续增长且 CPU > 85% 时扩容主线；临时突发或 CPU < 85% 时限流主线）

### 边界条件
- 熔断时间阶梯：3min 判断抖动，5min 进入评估，8min 无恢复必须触发  适用条件：下游错误率持续上升场景，主要针对非核心链路或已定义降级语义的链路 / 失效条件：强一致性链路（交易、支付、库存），降级安全前提不满足时该阈值不直接适用
- 降级响应安全判断：事前定义 70% 为主，现场校验 30% 为辅；无预案时默认明确失败  适用条件：所有需要熔断触发降级的服务 / 失效条件：无历史演练、无 Runbook 定义、降级与真实成功无区分标记的场景
- CPU 80% 为扩容评估线（压测来源），85% 为扩容动作线（事故来源）  适用条件：标准无状态服务，CPU 密集型处理节点 / 失效条件：I/O 密集型或内存密集型服务（CPU 不是性能瓶颈），以及有特定 GC 特征的 JVM 应用
- 流量持续性决定限流/扩容主从关系切换（持续增长→扩容主线；临时突发→限流主线）  适用条件：具备流量趋势监控的服务，CPU > 85% 场景 / 失效条件：无法区分流量持续性的场景（无监控或流量模式不确定），此时默认限流保护
- 可逆性切换阈值：效果差 <10 点选可逆，15-20 点重新评估，>20 点切不可逆（前提：故障仍在扩大）  适用条件：两个方案止血效果有量化差距的故障响应决策 / 失效条件：效果无法量化或两方案时间窗口差异极大时，该阈值不直接适用
- 慢查询介入阈值：P99 2x 为用户感知线，持续 10 分钟为毛刺过滤线，两者均满足才进入介入线  适用条件：数据库慢查询监控，以 P99 尾延迟为主要观测指标的场景 / 失效条件：P99 绝对值本身很低（如平均 10ms，2x 仍为 20ms）时，倍率指标可能误导；批量报表类查询（慢但不影响在线 P99）
- 大促前操作窗口三级：30min 只做止血开关，2h 低风险清理，6h+ 已验证修复（必须留 1-2h 观察窗口）  适用条件：有明确业务节点（大促、发布、高峰期）前的技术操作决策 / 失效条件：无可预期业务节点的持续高流量场景，大促前三级框架不适用
