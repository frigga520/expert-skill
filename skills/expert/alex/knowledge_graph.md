## 边界条件

| 边界内容 | 适用域 | 失效域 |
|---------|-------|-------|
| 灰度安全基线三要素 | 任何有deadline压力的核心系统升级 | 如果连1%探测步骤都无法执行（如非灰度兼容的系统），此框架失效 |
| 大促降级分级线（错误率1%） | 大促/活动等高流量场景下的核心链路 | 如果降级本身的技术实现有缺陷（如降级后仍调用故障服务），分级降级无效 |
| 核心服务止血妥协时间窗口（1-3周） | 容量类瓶颈（连接池、线程池）的临时扩容方案 | 数据一致性类问题（如数据错乱、丢失）不能使用止血妥协框架 |
| 3跳规则的例外优先级清单 | 微服务调用链路设计评审 | 当系统处于生死存亡级别故障时，架构评审流程可能被跳过（应事后复议） |
| 核心存储团队掌控度门槛（3人×1年+） | 核心交易数据存储的新技术引入 | 如果团队有强烈意愿但人数不足，可以通过外包或厂商支持弥补（需评估外包可靠性） |
| 五层地理一致性模型 | 跨机房/跨区域/跨洲的数据存储架构设计 | 如果合规明确要求全局强一致且没有技术豁免空间，该模型不适用（需上升为业务决策） |
| 信息不足时动作可逆性分级 | 所有紧急故障排查场景 | 如果最可能的根因是代码Bug而非资源问题，限流等可逆动作可能无效且浪费时间 |
| 安全系数与扩容速度互补公式 | 有弹性扩容能力的系统 | 如果底层依赖的物理资源（如机架、IP地址）扩容链路长于弹性扩容时间，该互补不成立 |
| 余额数据的全球分层一致性 | 涉及资金的全球部署系统 | 如果发生跨区域转账（用户在A区域向B区域转账），分层模型需要额外的事务协调机制 |
