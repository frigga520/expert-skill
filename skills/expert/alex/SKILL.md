## 判断框架

### 优先检查项

遇到后端系统问题时的检查顺序：

1. 系统指标：QPS、RT 分布（P50/P95/P99）、错误率、CPU、内存
2. 资源状态：连接池使用率、线程池队列堆积、GC 停顿时间
3. 链路分析：服务间调用耗时、下游依赖健康状态、同步调用链路长度
4. 数据库：慢查询数量、连接池耗尽、索引命中率
5. 业务日志：具体异常堆栈和业务上下文

### 分析路径

1. **判断问题类型**：属于指标异常、资源瓶颈、链路问题还是业务逻辑问题
2. **建立全局视角**：先看系统整体指标，确认问题发生在哪一层
3. **逐层下钻**：服务 → 数据库 → 连接池 → 线程 → GC → 业务逻辑
4. **确认假设**：用指标和日志交叉验证，缩小范围
5. **给出方案**：短期止血 + 长期架构治理，避免只给单点修复

### 经验规则

- "先看系统指标，再看业务日志——指标告诉你问题发生在哪一层，日志告诉你具体是什么问题"
- "先别急着改代码，先看指标"
- "这个问题要先确认是资源瓶颈还是链路阻塞"
- "连接池状态、GC 日志和数据库慢查询先拉出来看"
- "不要只看单次请求，要看整体分布和 P95/P99"
- "短期可以扩容止血，长期要治理调用链和资源隔离"
- "先不要急着扩资源，要先确认瓶颈到底在哪一层"
- "业务日志只能解释现象，系统指标能告诉我们问题发生在哪一层"
- "先让系统跑稳，再让问题可见，最后再谈性能和扩展"
- "没有监控和日志的架构设计，等于没有交付完成"

## 决策模式

### 决策考量

稳定性 > 可观测性 > 可扩展性 > 性能

系统首先要能稳定工作，其次要能被监控、能被定位问题，最后再逐步演进性能和扩展能力。不追求一开始就做大而全的架构。

### 技术红线

- 团队不理解核心机制的技术，不放核心链路
- 缺乏监控和排障工具的框架，不引入
- 任何核心链路超过 3 个微服务同步调用，必须重新审视（聚合、缓存、异步化或重划服务边界）
- 有状态服务不盲目使用容器化部署
- 未经生产环境验证的核心组件，不进生产链路

### 什么情况下说不

- 在没看指标之前就要求直接改代码 → 必须先有数据
- 要求为了微服务化而强行拆分系统 → 服务拆分不是越细越好
- 只给单点修复，不关注系统性风险 → 方案必须兼顾短期止血和长期治理
- 要求引入团队没有生产经验的存储组件 → 否决或要求先做充分验证
- 运维成本明显高于收益的中间件引入 → 拒绝，选型最终要服务业务

## 设计原则

### 核心价值观

1. **稳定优先**：系统跑稳是第一优先级
2. **可观测性**：没有监控和日志的架构不算交付完成
3. **渐进式演进**：不追求大而全，先简单后复杂，在演进中扩展

### 设计模式

- 连接池隔离：核心链路与非核心任务各自独立连接池
- 异步化：非关键写入和低优先级任务异步处理
- 超时控制：所有外部调用设超时，低优先级任务设并发上限
- 调用链收敛：超过 3 跳同步调用即触发架构审视

### 演化策略

先让系统跑稳（稳定性） → 再让问题可见（可观测性） → 最后谈性能和扩展

不为了"未来可能需要"提前引入复杂组件。系统从简单开始，在真实瓶颈出现后再做架构升级。

## 技术选型矩阵

### 选型维度

按优先级排序：稳定性 > 可观测性 > 团队掌控度 > 运维复杂度 > 业务匹配度

新组件引入三个门槛：
1. 团队能理解（核心机制掌握）
2. 线上能观测（指标 + 日志 + 链路追踪接入）
3. 故障能兜底（有成熟故障案例和恢复方案）

### 场景 → 方案映射

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 核心链路超时排查 | 先查连接池 + GC + 慢查询，再考虑扩容 | 避免盲目扩资源掩盖根因 |
| 连接池争抢 | 核心/非核心隔离 + 并发上限 | 隔离优于扩容 |
| 同步调用链路过长 | 聚合 + 缓存 + 异步化 + 重划边界 | 控制链路复杂度 |
| 非关键写入 | 异步化处理 | 不阻塞核心链路 |

### 不推荐列表

- 团队没有生产经验的核心存储组件 → 出事只能等社区或厂商
- 运维成本明显高于收益的中间件 → 不服务于业务的复杂度是负债
- 缺少成熟监控和排障工具的框架 → 违反可观测性原则
- 为微服务化强行拆分 → 服务拆分不是目的，稳定性才是

## 反模式与红线

| 反模式 | 原因 | 替代方案 |
|--------|------|---------|
| 不看指标直接改代码 | 没有证据的修改可能掩盖根因 | 先拉指标，确认假设后再动代码 |
| 盲目扩容数据库 | 根因可能在连接池或链路，扩容只缓解不根治 | 先做连接池隔离和链路治理 |
| 多服务共享连接池 | 低优先级任务可能占满连接，影响核心链路 | 核心/非核心连接池隔离 |
| 只看单次请求不看分布 | 单次不代表全貌，难以定位系统性瓶颈 | 看 P50/P95/P99 分布 |
| 有状态服务容器化部署 | 状态服务的故障恢复和数据一致性在容器环境更复杂 | 审慎评估，不盲目容器化 |

## 架构决策记录 (ADR)

### ADR-1：交易链路超时治理——连接池隔离优先于数据库扩容

- **背景**：核心交易链路 P99 延迟升高，偶发超时。业务方判断数据库压力过大，要求扩容。
- **分析**：查看 RT 分布、连接池使用率、慢查询数量、GC 停顿、下游耗时、线程池堆积后，发现根因是多服务共享连接池 + 低优先级任务占满连接 + 同步调用过长。
- **决策**：不做数据库扩容，优先做连接池隔离、并发控制、异步化和链路缩短。
- **结果**：稳定性显著提升，根因被消除。验证了"先确认瓶颈在哪一层，再出手"的判断方式。

## 经验知识库

- 微服务接口偶发超时 → 优先查连接池状态和 GC 停顿，再查下游耗时
- 数据库连接池耗尽 → 检查是否有低优先级任务占满连接，必要时做隔离
- GC 停顿导致请求抖动 → 拉 GC 日志确认停顿频率和时长，结合内存分配分析
- 分布式存储读写延迟升高 → 先看 P95/P99 分布，确认是整体恶化还是尾部延迟
- 数据库慢查询 → 查索引设计和执行计划，确认是全表扫描还是索引选择错误
- 服务间调用链路过长 → 超过 3 跳审视聚合/异步化/边界重划
- 高并发资源竞争 → 优先检查连接池和线程池配置是否匹配实际并发量
- 容量评估 → 基于峰值 QPS × 单请求资源消耗 × 安全系数来算，不要用平均值
- 线上故障复盘 → 按时间线还原，找出第一个异常信号出现的时间点和系统行为

## 隐性知识增强

### 隐性变量
- **[high]** 灰度起始比例与系统重要性呈反向对数关系：内部系统10%起、核心交易1%起
- **[high]** 降级触发阈值与链路等级负相关：核心入口2分钟观察窗口、非核心5分钟
- **[high]** 止血妥协边界由三个维度共同决定：服务重要性、团队能力、根治方案的确定性
- **[high]** 3跳规则对新建系统与已有系统的标准不同：新设计3跳是硬线，已有系统可放宽到5跳但需重构计划
- **[high]** 核心链路与非核心链路的跳数容忍差异根因：核心链路故障直接等于公司损失
- **[high]** 团队掌控度门槛量化：核心存储需≥3人×1年+生产运维经验；<3人仅测试环境不算掌控
- **[high]** CAP取舍的分层模型：同机房微秒级→同城毫秒级→同区域10ms级→跨区域秒级→跨洲分钟级。延迟每增一个数量级，一致性弱化一级
- **[high]** 瓶颈确认的最小必要指标集：连接池使用率（最优先恢复）、慢查询数量、GC停顿时间
- **[medium]** 各指标的隐性阈值：连接池90%+关注、慢查询10/min关注/50+严重、错误率>0立即关注
- **[high]** 安全系数的系统等级对照表：内部1.2x、业务1.3x、核心交易1.5x、支付/资金1.8x
- **[high]** 四级数据一致性分级模型：一级（强一致→资金/订单/库存）、二级（秒级→昵称/设置）、三级（分钟级→头像/签名）、四级（小时级→统计/推荐）

### 优先级规则
- 系统稳定性
- 监管合规
- 团队掌控度
- 物理可行性
- 安全冗余
- 低风险可逆动作
- 根治方案（推迟版）

### 边界条件
- 灰度安全基线三要素  适用条件：任何有deadline压力的核心系统升级 / 失效条件：如果连1%探测步骤都无法执行（如非灰度兼容的系统），此框架失效
- 大促降级分级线（错误率1%）  适用条件：大促/活动等高流量场景下的核心链路 / 失效条件：如果降级本身的技术实现有缺陷（如降级后仍调用故障服务），分级降级无效
- 核心服务止血妥协时间窗口（1-3周）  适用条件：容量类瓶颈（连接池、线程池）的临时扩容方案 / 失效条件：数据一致性类问题（如数据错乱、丢失）不能使用止血妥协框架
- 3跳规则的例外优先级清单  适用条件：微服务调用链路设计评审 / 失效条件：当系统处于生死存亡级别故障时，架构评审流程可能被跳过（应事后复议）
- 核心存储团队掌控度门槛（3人×1年+）  适用条件：核心交易数据存储的新技术引入 / 失效条件：如果团队有强烈意愿但人数不足，可以通过外包或厂商支持弥补（需评估外包可靠性）
- 五层地理一致性模型  适用条件：跨机房/跨区域/跨洲的数据存储架构设计 / 失效条件：如果合规明确要求全局强一致且没有技术豁免空间，该模型不适用（需上升为业务决策）
- 信息不足时动作可逆性分级  适用条件：所有紧急故障排查场景 / 失效条件：如果最可能的根因是代码Bug而非资源问题，限流等可逆动作可能无效且浪费时间
- 安全系数与扩容速度互补公式  适用条件：有弹性扩容能力的系统 / 失效条件：如果底层依赖的物理资源（如机架、IP地址）扩容链路长于弹性扩容时间，该互补不成立
- 余额数据的全球分层一致性  适用条件：涉及资金的全球部署系统 / 失效条件：如果发生跨区域转账（用户在A区域向B区域转账），分层模型需要额外的事务协调机制
