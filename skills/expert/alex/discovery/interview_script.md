## [开场] tg_001 — 灰度放量节奏

**领域背景**：微服务架构变更灰度发布场景

**控制说明**：A/B共享70%以上文本结构，只改变服务影响面。C叠加上线时间压力

> ⚠️ 请不要提前告知专家题目编号或 AB 结构

### Question A

你在负责一次MySQL 5.7到8.0的升级，涉及约500GB数据。服务是一个内部管理后台，仅公司20个运营人员使用，短暂不可用不影响外部用户。你怎么规划升级节奏——具体怎么放量、每步观察什么指标、什么情况需要回滚？

**主追问**：你刚才说的第一批流量比例和观察时间，是怎么确定的？

**备用追问**：
- 如果第一批完全正常，下一批你会加多少？
- 观察期间除了错误率，你还会看什么指标来判断能不能继续？
- 有没有哪个指标一旦异常你立刻回滚，其他指标可以容忍一阵？

**预期揭示**：
- 显性规则：灰度放量的标准步骤和默认观察指标
- 隐性变量信号：不同影响面下的默认灰度起始比例和加速节奏差异
- 优先级信号：内部工具场景下速度优先于谨慎的倾向

*（B 变化：服务影响面从内部管理后台（20人无外部影响）变为核心交易链路（500万笔/天，影响用户支付））*

### Question B

你在负责一次MySQL 5.7到8.0的升级，涉及约500GB数据。服务是核心交易链路入口，每天处理500万笔交易，任何中断都会影响用户支付和公司收入。你怎么规划升级节奏——具体怎么放量、每步观察什么指标、什么情况需要回滚？

**主追问**：和刚才那个内部后台的升级相比，你的策略有什么不同？

**备用追问**：
- 这个场景下第一批你会放多少流量？为什么选这个数？
- 每一步之间的观察期你会设多长？
- 如果第一批全绿，你会加速到什么节奏？

**预期揭示**：
- 显性规则：核心链路的灰度保守策略
- 隐性变量信号：影响面大小对灰度起始比例和观察窗口的隐性调节系数
- 优先级信号：交易场景下安全优先于速度，但存在一个最小必要速度的隐性底线

*（C 变化：沿用B的核心交易链路场景）*
*（冲突叠加：上线时间窗口从充裕变为硬性截止日期（周五前），且超期后果是大促可能崩溃，与灰度安全性冲突）*

### Question C

你在升级核心交易链路MySQL 8.0（500GB数据，500万笔/天）。业务VP要求必须本周五前完成——周六有大促活动，如果不升级数据库可能撑不住。今天周一。你怎么规划升级节奏和回滚策略？时间压力下哪些灰度步骤你会保留、哪些会压缩？

**主追问**：时间压力下，你会牺牲灰度策略的哪些部分？哪些是你死也不会动的？

**备用追问**：
- 如果最坏情况发生——灰度阶段发现问题严重——你会怎么做？
- 你说会压缩到X步——这算是接受了更高风险，还是找到了其他降低风险的手段？
- 大促崩溃风险和时间压力，哪个更严重？你怎么比较？

**预期揭示**：
- 显性规则：时间压力下的灰度策略调整
- 隐性变量信号：灰度安全底线——哪些灰度步骤即使在极端压力下也不能取消
- 优先级信号：时间压力和大促风险之间的隐性优先级排序

## [核心] tg_003 — 止血妥协边界

**领域背景**：系统容量瓶颈场景

**控制说明**：A/B共享70%以上文本结构，只改变服务类型。C叠加团队能力约束

> ⚠️ 请不要提前告知专家题目编号或 AB 结构

### Question A

商品推荐服务的数据库连接池在高峰期频繁打满（使用率95%+），导致偶尔超时。方案A是立刻把连接池从200扩到400（1天上线），方案B是做读写分离+核心/非核心连接池隔离+异步化改造（预估2周）。A只是暂缓问题但快速，B根治但需要时间。你选哪个？为什么？

**主追问**：你选A还是B——判断的核心标准是什么？

**备用追问**：
- 如果选A，这个临时方案你能接受维持多久？什么信号触发你启动B？
- 如果选B，这两周内服务继续偶尔超时，你怎么和业务方交代？
- 有没有第三个选项——部分B再加A兜底？

**预期揭示**：
- 显性规则：对非核心服务接受临时止血方案的条件
- 隐性变量信号：服务重要性如何改变妥协意愿——隐性边界在哪里
- 优先级信号：非核心场景下速度优先于根治的倾向

*（B 变化：服务从商品推荐（非核心）变为支付服务（核心交易，超时直接影响用户支付和公司收入））*

### Question B

支付服务的数据库连接池在高峰期频繁打满（使用率95%+），导致偶尔超时。方案A是立刻把连接池从200扩到400（1天上线），方案B是做读写分离+核心/非核心连接池隔离+异步化改造（预估2周）。A只是暂缓问题但快速，B根治但需要时间。你选哪个？为什么？

**主追问**：和刚才推荐服务的场景相比，你的选择变了吗？为什么？

**备用追问**：
- 如果选A，支付服务用临时方案撑2周你能接受吗？容忍度和推荐服务一样吗？
- 如果选B，这两周内每超时一次都可能损失订单——这个风险你扛吗？
- 推荐服务和支付服务在'可接受临时方案'上有多大区别？

**预期揭示**：
- 显性规则：核心服务对临时止血方案的零容忍倾向
- 隐性变量信号：止血妥协的隐性边界——服务等级如何切换'允许临时方案'到'必须根治'
- 优先级信号：支付场景下根治优先于速度的倾向

*（C 变化：沿用B的核心支付服务场景）*
*（冲突叠加：团队能力约束——唯一有经验的工程师不在，根治方案的质量风险急剧增加，与'核心服务必须根治'原则冲突）*

### Question C

支付服务连接池打满偶尔超时，方案A扩容1天，方案B架构改造2周。但目前团队中唯一熟悉连接池隔离改造的工程师正在异地出差做另一个紧急项目，至少3周后才能回来。如果强行做方案B，只能让不熟悉这块的人来，质量风险更高。你怎么选？

**主追问**：团队能力的约束有没有改变你对A和B的判断？

**备用追问**：
- 如果你选A（扩容），这和推荐服务上的选择本质一样——那核心服务不能妥协的原则还成立吗？
- 如果让不熟悉的人做B，引入了新Bug导致更严重事故——和方案A的超时风险比，哪个更糟？
- 如果选B但让异地工程师远程指导——这种半掌控状态你能接受吗？

**预期揭示**：
- 显性规则：团队能力约束下的方案选择调整
- 隐性变量信号：核心服务必须根治原则的失效条件——什么情况下Alex会放宽标准
- 优先级信号：架构原则 vs 现实约束的隐性取舍机制

### Question A

**主追问**：这个4跳的设计你怎么看——能接受吗，还是需要改？

**备用追问**：
- 如果可以保留4跳，你要求加什么保障措施？
- 如果要求你缩短到3跳以内，你会砍掉哪一跳？为什么？
- 你说这个场景可以接受——如果换成用户下单场景，同样4跳你还会接受吗？

**预期揭示**：
- 显性规则：3跳规则在不同链路等级下的适用标准
- 隐性变量信号：3跳规则生效域的隐性边界——非核心链路的跳数容忍上限
- 优先级信号：非核心链路中简洁性 vs 业务灵活性的权衡

*（B 变化：链路从用户画像聚合（非核心推荐）变为订单创建（核心交易，涉及支付和收入））*

### Question B

在架构评审会上，团队设计了一个订单创建服务：API网关 → 订单服务 → 库存扣减服务 → 支付验证服务 → 风控引擎，总共4个同步调用跳。这个链路每天处理约100万笔订单，涉及用户支付和公司收入。你作为评审人，对这个设计有什么意见？

**主追问**：这个订单链路的4跳设计和刚才推荐链路的4跳——你的判断一样吗？

**备用追问**：
- 你说必须改——改到什么程度你才批准？最多接受几跳？
- 如果业务方告诉你这4步缺一不可且无法异步化——你怎么回应？
- 你说核心链路标准不同——能具体说不同在哪里吗？

**预期揭示**：
- 显性规则：核心链路对调用跳数的硬性约束
- 隐性变量信号：3跳规则从审视到禁止的隐性升级条件——什么情况下3变2，什么情况下4可接受
- 优先级信号：核心链路中简洁性的绝对优先级

*（C 变化：沿用B的核心交易链路场景）*
*（冲突叠加：监管合规的硬性要求（不能不做）vs 已有系统的重构成本（3年老链路），与缩短调用链的架构原则冲突）*

### Question C

订单创建链路已在线3年，4个同步调用跳：网关→订单→库存→支付→风控。现在有监管合规新需求，必须再加一个合规检查服务变成5跳。团队说合规要求不能不做，且已有链路3年了很难重构。你怎么处理？

**主追问**：监管要求和架构原则冲突的时候，你怎么处理？

**备用追问**：
- 你说可以接受5跳因为合规不得不加——那'不得不'的判断标准是什么？
- 如果是业务方自己提的需求而非监管要求，同样的5跳你会接受吗？
- 你说先接受但要规划重构——重构的时间窗口是多久？如果一直没排上期呢？

**预期揭示**：
- 显性规则：已有系统与新建系统在架构约束上的区分标准
- 隐性变量信号：3跳规则的终极例外清单——什么条件可以打破（监管、存量成本？优先级排序？）

*（C 变化：沿用B的核心订单存储场景）*
*（冲突叠加：沉没成本压力——3个月的TiDB POC投入 vs 换回MySQL的浪费，与团队掌控度原则冲突）*
